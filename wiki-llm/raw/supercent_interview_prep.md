# 슈퍼센트 AI Application Engineer — 1차 면접 준비

---

## 1분 자기소개 스크립트 (55~60초)

저는 **"만들어서 증명하는"** 개발자 임재준입니다.

프론트엔드로 시작했지만, 문제 앞에서 기술의 경계를 그어본 적이 없습니다. 산업용 3D 프린터 수십 대를 실시간으로 관제하는 C-Hub에서 프론트엔드 전체를 맡아 기획부터 배포까지 주도했고, 그 결과가 GS인증 1등급으로 돌아왔습니다. 현재 국내와 필리핀 현장에서 실제로 운영 중입니다.

지금은 AI 엔지니어링으로 스펙트럼을 넓히고 있습니다. 이번 과제에서 UA 마케터가 경쟁 게임 트렌드를 파악하고 광고 소재를 기획하는 데 걸리는 시간을 줄이고 싶었습니다. Google Play 실시간 차트 수집부터 Claude 4단계 앙상블 분석, Vision 스크린샷 분석, 광고 소재 6종 생성까지 — 장르명 하나 입력하면 끝나는 파이프라인을 만들었습니다.

과제를 제출하고 나서도 멈추지 않았습니다. 토큰 비용을 89% 절감하는 구조 개선, iOS 데이터 통합, 피드백 루프까지 8가지를 고도화했고, 슈퍼센트 실제 게임 26개 타이틀의 앱스토어 리뷰를 분석하는 Centiment도 별도로 만들었습니다.

저는 AI를 **"쓰는"** 개발자가 아니라, AI로 조직이 체감할 수 있는 변화를 만드는 엔지니어가 되고 싶습니다. 그리고 슈퍼센트야말로 그 비전을 실현할 수 있는 최적의 무대라고 확신합니다.

## 예상 질문 & 답변 전략

### 1. 지원 동기

---

**Q. 왜 이직하려 하나요? 현재 회사에서 계속하면 안 되나요?** `필수`

C-Hub는 산업용 솔루션이라 AI를 적용할 수 있는 도메인이 제한적입니다. 저는 지금 AI 엔지니어링으로 방향을 잡고 있는데, 실제로 수억 명의 유저 데이터와 게임 시장 데이터가 있는 환경에서 AI를 적용하는 것과 개인 프로젝트로 만드는 것은 완전히 다른 경험이라고 생각합니다.

슈퍼센트는 누적 15억 다운로드의 실제 데이터가 있고, UA 마케팅부터 게임 제작까지 AI를 연결할 수 있는 접점이 많습니다. 제가 과제에서 만든 Centinel과 Centiment가 실제 내부 데이터와 결합하면 어떤 임팩트가 나오는지 직접 경험하고 싶습니다.

> ⚠️ 현 회사를 부정적으로 말하지 마세요. "현 회사가 나쁜 게 아니라 내가 가고 싶은 방향이 명확해진 것"으로 프레이밍하세요.

---

**Q. 왜 슈퍼센트인가요? 다른 회사도 있을 텐데요.** `필수`

하이퍼캐주얼 게임 시장은 데이터 싸움입니다. UA 마케팅, 소재 A/B 테스트, 차트 트렌드 분석 — 이 모든 게 AI가 가장 잘 할 수 있는 영역과 정확히 겹칩니다. 그리고 슈퍼센트는 국내 앱 퍼블리셔 1위, 글로벌 5위로 그 데이터가 실제로 있는 회사입니다.

과제를 준비하면서 슈퍼센트의 게임들이 글로벌 차트에서 어떻게 움직이는지 직접 추적했습니다. 그 과정에서 이 도메인에서 AI 엔지니어가 만들 수 있는 임팩트의 크기를 체감했습니다. 단순히 AI를 쓰는 회사가 아니라, AI로 게임 산업 자체를 바꾸려는 방향이 저와 맞습니다.

---

**Q. AI Application Engineer라는 직무가 생소한데, 어떻게 알게 됐나요?** `심화`

사실 제가 해온 것들을 돌아보면 이 직무에 자연스럽게 수렴해 왔습니다. C-Hub에서 복잡한 실시간 문제를 풀면서 기술 범위를 계속 넓혔고, 개인 프로젝트에서 AI API를 연동하면서 "AI를 서비스에 녹이는 것"이 제가 가장 몰입하는 작업이라는 걸 알았습니다. 슈퍼센트 공고를 보고 이 직무가 제가 가려는 방향을 정확히 정의하고 있다는 걸 느꼈습니다.

### 2. 파이프라인 설계

---

**Q. 왜 4단계 앙상블인가요? 오버엔지니어링 아닌가요?** `핵심`

단일 모델 단일 호출은 하나의 관점만 나옵니다. Stage 1에서 트렌드 분석가/Vision/왜인기인가를 병렬로 돌리는 건, 같은 게임 데이터를 세 전문가 시각으로 동시에 보는 구조입니다. Stage 2에서 분석형 A와 창의형 B를 병렬로 돌리는 건, 동일 입력에서 다른 방향의 인사이트를 뽑아 최종 선별 후보군을 넓히는 목적입니다.

**비용 반박이 오면:** UA 마케터가 소재 하나 기획하는 시간 대비 API 비용은 미미합니다. 단일 호출로 품질이 낮아 소재를 다시 만들어야 하는 비용이 더 큽니다.

---

**Q. Vision 결과 cascade injection이 실제로 품질 차이를 만드나요?** `핵심`

있었습니다. Vision 없이 생성한 소재는 텍스트 메타데이터 기반의 추상적인 카피였습니다. Vision을 넣으면 '탭 기반 조작, 네온 색상, 폭발 이펙트'처럼 실제 스크린샷에서 추출된 시각 요소가 카피에 반영됩니다. UA 소재는 영상/이미지와 함께 나가는데, 시각 레퍼런스가 없는 카피와 있는 카피의 방향성이 달라집니다.

---

**Q. Haiku × 2 병렬 앙상블에서 같은 모델을 두 번 쓰는 이유가 뭔가요?** `심화`

같은 모델이더라도 시스템 프롬프트를 다르게 주면 전혀 다른 방향의 결과가 나옵니다. 분석형 A는 데이터 기반으로, 창의형 B는 마케팅 감각으로 접근하도록 지시했습니다. 두 결과를 모아 Sonnet이 선별하면 단일 방향보다 더 넓은 후보군에서 고르는 구조가 됩니다. 비용은 Haiku라 낮고, 다양성 확보 효과는 확실합니다.

---

**Q. FOMO 편향을 발견한 과정이 어떻게 됐나요?** `심화`

실제로 여러 장르를 돌려보다가 매번 FOMO 톤이 expectedCTR 1위로 나오는 걸 발견했습니다. 처음엔 "FOMO가 원래 잘 되는 거 아닌가" 싶었는데, 장르나 게임이 달라도 패턴이 고정된다는 게 이상했습니다. AI가 "FOMO = 높은 CTR"이라는 스테레오타입을 학습한 것이었습니다. CTR 기준 선별을 없애고 hook+mainCopy 품질 기반으로 바꿨고, CTR 점수 분산 지시도 추가했습니다.

---

**Q. 피드백 루프 설계에서 글로벌 집계와 개인화를 분리한 이유가 뭔가요?** `심화`

로그인 없이 먼저 데이터를 쌓는 게 우선이라고 판단했습니다. 개인화는 user_id만 채우면 되도록 스키마를 설계해뒀습니다. 지금 당장 로그인 시스템을 붙이면 진입 장벽이 생겨 피드백 데이터 자체가 안 쌓입니다. 글로벌 집계로 먼저 패턴을 확보하고, 사용자가 늘면 개인화로 전환하는 단계적 전략입니다.

### 3. 기술 선택

---

**Q. Haiku랑 Sonnet을 어떻게 나눠 쓴 건가요?** `핵심`

작업 성격으로 나눴습니다. 판단/선별/종합이 필요한 곳엔 Sonnet, 패턴 인식/분류/반복 작업엔 Haiku입니다. 트렌드 분석과 최종 오케스트레이터는 Sonnet, 왜인기인가 개별 분석과 앙상블 생성은 Haiku입니다. Vision은 멀티모달 품질이 중요하니 Sonnet입니다. 이 기준으로 비용을 최소화하면서 품질이 필요한 지점에만 Sonnet을 썼습니다.

---

**Q. 왜 WebSocket 아니고 SSE인가요?** `핵심`

이 파이프라인은 서버 → 클라이언트 단방향입니다. 요청은 클라이언트가 한 번 보내고, 이후엔 서버가 단계 완료마다 결과를 밀어주는 구조입니다. 양방향 통신이 필요 없으니 WebSocket의 복잡한 연결 관리가 불필요합니다. SSE는 HTTP 위에서 동작하고 Next.js ReadableStream으로 구현이 단순하며 Vercel 환경과도 호환됩니다.

---

**Q. Vision 캐싱에서 TTL 없이 URL 기반으로 무효화한 이유가 뭔가요?** `심화`

앱 스크린샷 URL은 앱 업데이트 시 바뀝니다. URL이 바뀌면 캐시 키가 자동으로 달라지니 별도 TTL 없이 자연스럽게 무효화됩니다. TTL을 쓰면 URL이 안 바뀌어도 주기적으로 재분석이 트리거되어 불필요한 Sonnet 호출이 발생합니다. 다만 URL이 같은데 이미지만 교체되는 경우는 감지 못하는 한계가 있고, 이는 수동 재분석으로 대응하도록 설계했습니다.

---

**Q. google-play-scraper 말고 공식 API를 쓰지 않은 이유가 있나요?** `심화`

Google Play의 공식 차트 API는 개발자 콘솔 앱 등록 없이는 접근이 불가능하고, Sensor Tower 같은 상용 API는 유료입니다. 과제 범위에서 빠른 프로토타이핑을 위해 스크래퍼를 선택했습니다. 비공식 스크래퍼라 Google 정책 변경 시 깨질 수 있다는 리스크는 인지하고 있습니다. 프로덕션이라면 Sensor Tower나 공식 API로 교체하는 게 맞습니다.

---

**Q. Next.js App Router를 선택한 이유가 있나요?** `심화`

API Routes와 프론트엔드를 하나의 레포에서 관리할 수 있고, Vercel 배포와 Cron이 네이티브로 통합됩니다. SSE 스트리밍도 ReadableStream으로 바로 구현 가능합니다. 과제처럼 빠르게 풀스택을 혼자 만들어야 하는 상황에서 배포와 인프라 관리 비용을 최소화하는 선택이었습니다.

---

### 4. 백엔드 역량

---

**Q. 백엔드 실무 경험이 없는데 괜찮을까요?** `핵심`

백엔드를 전담한 실무 경험은 없습니다. 다만 Centinel에서 SSE 스트리밍 서버, PostgreSQL 스키마 설계, Vercel Cron 운영까지 혼자 했습니다. GeckoHub에서는 Django DRF REST API와 NextAuth JWT 인증을 통합했고, 프리랜서 작업에서 GCP Cloud Run에 Node.js 서버를 올리고 BigQuery에 4천만 건을 적재했습니다.

전문 백엔드 개발자 수준의 깊이는 아니지만, 서비스를 끝까지 만드는 데 필요한 범위는 직접 해왔습니다. 이 직무가 요구하는 것도 백엔드 전문성보다 "서비스 전반을 이해하고 만드는 감각"이라고 이해하고 있습니다.

---

**Q. DB 설계 결정 근거를 설명해주실 수 있나요?** `심화`

두 가지가 핵심이었습니다. 첫째, analysis_history에 games/insight/ad_copies를 jsonb로 저장한 이유는 분석 결과 구조가 파이프라인 개선에 따라 자주 바뀌기 때문입니다. 컬럼으로 고정하면 스키마 마이그레이션이 반복됩니다. 둘째, chart_snapshots는 fetched_at 타임스탬프 기반으로 최신 vs 이전 스냅샷을 비교해 rankChange를 계산합니다. 별도 diff 테이블을 두지 않고 쿼리로 해결해 스키마를 단순하게 유지했습니다.

---

**Q. 인프라나 DevOps 경험은 어느 수준인가요?** `심화`

Vercel 배포, Cron 스케줄러, Supabase 운영은 직접 했습니다. GCP Cloud Run + Cloud Scheduler로 서버리스 파이프라인을 구성한 경험도 있습니다. Docker나 Kubernetes는 실무 경험이 없습니다. 다만 이번 과제에서도 Vercel Cron을 처음 써봤지만 하루 안에 스냅샷 시스템까지 구현했습니다. 필요한 범위가 생기면 빠르게 흡수하는 게 저의 방식입니다.

---

### 5. 프로덕션 레벨

---

**Q. 이걸 슈퍼센트 내부 툴로 쓴다면 뭐가 문제가 될까요?** `핵심`

세 가지를 꼽겠습니다.

첫째, Rate Limit 핸들링입니다. 현재 API 에러 시 단순 throw인데, 프로덕션에선 exponential backoff 재시도 로직이 필요합니다.

둘째, google-play-scraper 의존성 리스크입니다. 비공식 스크래퍼라 Google 정책 변경 시 즉시 깨집니다. Sensor Tower 같은 공식 API로 교체해야 합니다.

셋째, Vision 비용 예측 불가입니다. 스크린샷 수에 따라 입력 토큰이 크게 달라지는데 현재 사용량 모니터링이 없습니다. 토큰 예산 상한과 알림이 필요합니다.

---

**Q. 동시에 여러 명이 분석 요청하면 어떻게 되나요?** `심화`

현재는 요청별로 독립적인 API 호출이 발생해 동시 요청이 많아지면 Anthropic API Rate Limit에 걸릴 수 있습니다. 프로덕션에서는 요청 큐를 두고 순차 또는 제한된 병렬로 처리하는 구조가 필요합니다. Vision 캐싱은 이미 구현되어 있어 동일 게임 세트 재분석은 API 호출 없이 바로 반환됩니다.

---

**Q. AI 응답 품질이 일관되지 않을 때 어떻게 처리했나요?** `심화`

두 가지로 접근했습니다. 첫째, 출력 형식을 JSON으로 강제하고 파싱 실패 시 graceful degradation으로 처리했습니다. Vision 분석 실패 시 null을 반환하고 이후 단계는 Vision context 없이 진행하는 방식입니다. 둘째, Haiku 배치 분류에서 반환 건수가 입력과 다를 때 neutral/other/[]로 자동 패딩하는 방어 로직을 넣었습니다. 완벽한 응답을 기대하기보다 실패를 예상하고 설계했습니다.

---

**Q. Vercel Cron이 실패하면 스냅샷 데이터가 누락되는데 어떻게 처리했나요?** `심화`

수동 저장 버튼을 별도로 구현해뒀습니다. Cron이 실패하더라도 관리자가 직접 스냅샷을 트리거할 수 있습니다. Rising 탭은 최신 vs 1시간 이상 전 스냅샷 비교로 동작하기 때문에, 스냅샷이 2개 이상 쌓여야 작동한다는 것을 README에도 명시했습니다. 프로덕션에서는 Cron 실패 시 Slack 알림 연동이 필요합니다.

### 6. 직무 적합성

---

**Q. AI Application Engineer로서 본인의 역할을 어떻게 정의하나요?** `핵심`

코드를 잘 짜는 것보다 조직이 변화를 체감하게 만드는 것이라고 생각합니다. Centinel을 만들면서 가장 집중한 건 UA 마케터가 실제로 쓸 수 있는가였습니다. 기술적으로 흥미로운 파이프라인보다, 장르명 하나 입력해서 광고 소재 6종이 나오는 흐름이 실제 업무에 연결되는지를 더 고민했습니다.

**Q. 특정 기술 깊이보다 넓은 기술 감각을 중시한다고 했는데, 본인은 어떻게 생각하나요?** `핵심`

동의합니다. 저는 React/TS가 주력이지만 이번 과제에서 스크래퍼 서버, DB 설계, Cron 인프라, Vision 멀티모달까지 혼자 붙였습니다. GeckoHub에서는 Django를 처음 썼고, 광고 파이프라인 프리랜서에서는 GCP를 처음 썼습니다. 목표가 명확하면 모르는 기술을 빠르게 흡수하는 게 저의 패턴입니다. 다만 넓이만 있고 깊이가 없으면 설계 판단이 틀릴 수 있어서, AI 오케스트레이션 영역만큼은 깊이를 계속 쌓고 있습니다.

**Q. Centiment는 왜 만든 건가요?** `심화`

Centinel을 만들다 보니 자연스럽게 연결되는 그림이 보였습니다. 광고 소재를 잘 만들려면 경쟁 시장만이 아니라 우리 게임 유저가 지금 뭘 좋아하고 싫어하는지도 알아야 합니다. 슈퍼센트 게임 26개 타이틀의 셀링포인트를 뽑아서 Centinel 소재 생성에 주입하면, 실제 유저 언어 기반 광고 카피 자동화가 됩니다. 과제 범위를 넘은 게 아니라 과제가 어디로 연결되는지 끝까지 생각한 결과입니다.

**Q. AI 발전 속도를 따라가는 본인만의 방법이 있나요?** `심화`

읽기만 하면 금방 잊어버립니다. 새로운 AI 기술이 나오면 바로 뭔가를 만들어봅니다. Claude API를 처음 접한 후 과제를 통해 멀티모달 앙상블까지 구현한 게 그 패턴입니다. 또 토큰 비용, 응답 구조, 실패 케이스 같은 "실제로 쓸 때 맞닥뜨리는 문제"는 직접 만들어봐야만 체득됩니다. 개인 프로젝트가 저의 AI 러닝 환경입니다.

**Q. 비개발 직군(영업, 전기공사)을 거친 게 개발에 어떤 영향을 줬나요?** `심화`

비즈니스 요구사항과 기술의 연결 고리를 먼저 이해하게 됐습니다. IT 영업에서 제안서를 쓰면서 "이 기능이 왜 필요한가"를 사용자 입장에서 먼저 생각하는 습관이 생겼습니다. Centinel에서도 기술 파이프라인보다 UA 마케터가 실제로 쓸 수 있는가를 먼저 고민한 게 이 배경에서 나온 것 같습니다. 현장에서 팀 리더로 일하면서 커뮤니케이션과 책임감도 단단해졌습니다.
